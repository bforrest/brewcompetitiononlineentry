# Phase 3.7: Registration Extraction — Design

**Date:** 2026-07-22
**Context:** Phase 3.1–3.4 extracted Entry, Judging, AdminPreferences, and Export into `src/Domain/`. This is the first Phase 3.7 workflow: entrant self-registration. Unlike 3.1–3.4, this workflow ships with a working dual-path (legacy vs. modern) equivalence proof from day one, at the user's explicit request — 3.3/3.4 shipped "COMPLETE" docs that a same-day trust audit found were false (see `Docs/PHASE_3_TRUST_AUDIT.md`); this design exists partly to not repeat that.

---

## Prerequisite: E2E infrastructure fixes (already applied)

Before any registration work could be verified end-to-end, two blockers had to be cleared:

1. **Stale container bind mount, not a real HTTPS/TLS bug.** `TESTING_HEALTH_DASHBOARD.md` claimed the entire E2E tier was broken by a missing TLS listener. It wasn't — `docker/.htaccess` already disables the HTTPS redirect for local dev, and `docker-compose.yml` already bind-mounts it over the root `.htaccess`. The running container just predated that fix and needed `docker compose up -d --force-recreate` (the exact gotcha documented from Phase 0). No code change was needed; `smoke.spec.ts` went from 3/4 to passing once recreated (4th failure below).

2. **`judging_locations` seed date silently force-closes registration.** `includes/constants.inc.php:262-265` sets `registration_open = 2` (closed) and `entry_window_open = 2` whenever *any* `judging_locations` row's `judgingDate` is in the past — regardless of `contest_info`'s deadlines. `docker/02-open-registration.sql` already pushes the contest deadlines to 2030 for this exact reason, but never touched `judging_locations`, whose one seeded row had a stale date. **Fixed:** extended that same SQL file to also push `judgingDate`/`judgingDateEnd` to 2030 (commit pending). This had been silently breaking `registerEntrant()` — the shared e2e helper used by `entrant-journey.spec.ts` and `dual-path-verification.spec.ts` — the entire time; the HTTPS bug just prevented anyone from reaching far enough to notice.

With both fixed, all 4 `smoke.spec.ts` tests pass, including fresh entrant registration.

**Known, separate, out-of-scope bug found in the process:** with both blockers cleared, `entrant-journey.spec.ts`'s "modern routes" test now reaches `/entries/my` and gets a 500: `RuntimeException: mysqli connection not initialized in $GLOBALS['connection']` ([container.php:125](../../src/Kernel/container.php#L125)). `index.php` deliberately never eagerly bootstraps `paths.php`/`site/config.php` (a documented Task 9 decision — legacy routes lazily self-bootstrap), but the modern `EntryController` routes never bootstrap it either. This is a real, previously-hidden gap in the **Entry** domain's HTTP wiring, unrelated to registration. Tracked, not fixed here, per user decision.

## Scope

**In scope:** entrant self-registration — `?section=register&go=entrant`, `filter=default` branch of `process_users_register.inc.php`. Includes the judge/steward/staff opt-in checkboxes that appear on the entrant form itself (they set `staff_judge`/`staff_steward`/`staff_staff`).

**Out of scope (later increment):** admin-driven account creation (`filter=admin`), the dedicated `go=judge`/`go=steward` registration entry points, already-logged-in re-registration edge cases.

**Tables written:** `users`, `brewer`, `staff` (insert-or-update). **External calls:** reCAPTCHA/hCaptcha verification, PHPMailer confirmation email (conditional on `prefsEmailRegConfirm`).

---

## Domain architecture

New `src/Domain/Registration/`, mirroring the Entry domain's shape (chosen as the reference because it's the domain the trust audit most thoroughly fixed and re-verified):

- **`Command\RegisterEntrantCommand`** — hydrated from the raw POST array, mirrors `CreateEntryCommand`.
- **`ValueObject\Email`** — format validation + lowercase normalization (matches legacy's `FILTER_SANITIZE_EMAIL`).
- **`ValueObject\RegistrantId`** — the new `users.id`/`uid`. Deliberately a fresh value object scoped to this domain rather than reusing `Entry\ValueObject\BrewerId`, to avoid a cross-domain dependency in the wrong direction.
- **`Repository\RegistrationRepository`** — `Connection`-based (prepared statements): `emailExists(Email): bool`, `insertUser(...): RegistrantId`, `insertBrewerProfile(...)`, `upsertStaffRow(...)` (mirrors legacy's check-then-insert-or-update).
- **`Service\RegistrationService`** — orchestrates: registration-window check → duplicate-email check → CAPTCHA verification → password hash (`password_hash`/BCRYPT, unchanged from legacy) → security-question hash (phpass, unchanged — needed for hash-format parity) → the 3 DB writes → optional confirmation email → returns `RegistrantId`.
- **`Exception\`**: `RegistrationClosedException`, `DuplicateEmailException`, `CaptchaVerificationFailedException`.

**CAPTCHA:** behind a `CaptchaVerifier` interface — a real implementation (ports the existing reCAPTCHA/hCaptcha HTTP-call logic verbatim) and a `NullCaptchaVerifier` (always succeeds). DI binds by `prefsCAPTCHA`, matching legacy's own bypass and `docker/03-e2e-fixtures.sql`'s existing `prefsCAPTCHA=0`.

## HTTP layer

Following the `EntryController` pattern:

- `GET /register` → render the modern registration form.
- `POST /register` → `RegistrationController::postRegister()` builds `RegisterEntrantCommand`, calls `RegistrationService::register()`, catches domain exceptions → HTTP status + re-rendered form with errors (same pattern as `EntryController::postCreate`).

## Session establishment

Traced how legacy actually logs a fresh registrant in: `process_users_register.inc.php`'s success path sets **only** `$_SESSION['loginUsername']` + `csrf_token_generate(true)`, then redirects. It does **not** set `$_SESSION['userLevel']` — that's set by `includes/logincheck.inc.php:118` on the *next* page load's bootstrap, explicitly commented "Task 10 fix" (the same Phase 2 regression already in memory, where fresh logins were locked out because `userLevel` was never set).

**Decision:** the modern controller does not reinvent session logic. It writes the identical `loginUsername` + CSRF-regen, then redirects (302) to `/entries/my`, relying on `AuthenticationMiddleware` (the already-fixed, already-proven modern equivalent of `logincheck.inc.php`) to hydrate `userLevel`/identity on the next request — exactly mirroring legacy's own redirect-then-rehydrate behavior instead of re-deriving it and risking the same class of bug again.

## Dual-path verification

**1. Integration test (PHPUnit, runs today, no browser needed).** `RegistrationDualPathTest`: identical input driven through the legacy processor logic and through `RegistrationService::register()` against the same transactional test DB. Asserts: same `userLevel`/`userAdminObfuscate`, a password hash that verifies against the same plaintext (bcrypt/phpass are salted, so not byte-identical), identical `brewer` field values (modulo the auto-increment id), identical `staff` row state for the same opt-in flags. This is the fast, deterministic, CI-friendly gate — the thing 3.2–3.4 skipped.

**2. Playwright spec** (`registration-dual-path.spec.ts`), following `export-dual-path.spec.ts`'s structure: `register via legacy form` / `register via modern form`, each verifying login success, plus a third test asserting equivalent DB state (matching `dual-path-verification.spec.ts`'s existing pattern for entries).

**Verification discipline** (trust audit recommendation #7): before any doc in this work claims "complete," actually run `./vendor/bin/phpunit --testsuite=Unit`, the integration tier, `./vendor/bin/phpstan analyse`, and the new Playwright spec, and report real numbers.

## Open questions / risks carried into implementation planning

- Exact mechanism for porting the reCAPTCHA/hCaptcha HTTP-call code into `CaptchaVerifier` without duplicating logic that might drift from the legacy version (candidate: legacy keeps calling a shared verification function both paths use).
- Whether the confirmation email should be sent via a new `Registration`-domain service wrapping `email.lib.php`, or by calling the existing `sendPHPMailerMessage` helper directly — no need to re-architect email infra for this pass.
- `EntryService::delete()`'s missing-validation gap (trust audit's still-open item) and the Entry `/entries/my` mysqli bug found above are both explicitly **not** part of this design; tracked separately.
