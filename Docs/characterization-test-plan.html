1. What Are Characterization Tests?

A characterization test doesn't assert what the code should do — it records what the code currently does. The goal is to create a safety net: if you refactor a function and the test still passes, you know you haven't changed any observable behavior. If the test fails, the diff tells you exactly what shifted.

This matters enormously for common.lib.php specifically. It has 108 functions, 5,477 lines, and has been touched in 229 commits — any change without tests risks a silent regression that could affect fee calculations, judging scores, or registration flows across the entire application.

The golden rule of characterization testing Run the existing code with known inputs, capture the output, and hard-code that output as the expected value. You are not deciding whether the output is "correct" — you're documenting what it is.

// tests/Unit/ConversionFunctionsTest.php
<?php
use PHPUnit\Framework\TestCase;

// Load just the functions we need
require_once __DIR__ . '/../../lib/common.lib.php';

class ConversionFunctionsTest extends TestCase
{
    /**
     * Characterization: temp_convert with "F" flag
     * actually converts Fahrenheit → Celsius
     * (despite the inline comment claiming otherwise).
     */
    public function test_temp_convert_F_flag_converts_fahrenheit_to_celsius(): void
    {
        // Capture current behavior — do NOT "fix" values
        $this->assertSame(0.0,   temp_convert(32, "F"));
        $this->assertSame(100.0, temp_convert(212, "F"));
        $this->assertSame(-17.8, temp_convert(0, "F"));
        $this->assertSame(-40.0, temp_convert(-40, "F"));
    }

    public function test_temp_convert_C_flag(): void
    {
        $this->assertSame(0.0,   temp_convert(32, "C"));
        $this->assertSame(100.0, temp_convert(212, "C"));
    }

    public function test_weight_convert_all_paths(): void
    {
        $this->assertSame(2.2,   weight_convert(1, "pounds"));
        $this->assertSame(0.04,  weight_convert(1, "ounces"));
        $this->assertSame(28.35, weight_convert(1, "grams"));
        $this->assertSame(0.45,  weight_convert(1, "kilograms"));
    }

    public function test_addOrdinalNumberSuffix(): void
    {
        $this->assertSame("1st",  addOrdinalNumberSuffix(1));
        $this->assertSame("2nd",  addOrdinalNumberSuffix(2));
        $this->assertSame("3rd",  addOrdinalNumberSuffix(3));
        $this->assertSame("11th", addOrdinalNumberSuffix(11));
        $this->assertSame("12th", addOrdinalNumberSuffix(12));
        $this->assertSame("13th", addOrdinalNumberSuffix(13));
        $this->assertSame("21st", addOrdinalNumberSuffix(21));

    public function test_currency_info_method_1_usd(): void

4. Tier 2 — Database-Dependent Functions With Seams

Medium — Requires Test Database

54 functions in common.lib.php make direct mysqli_query() calls. They can't be unit-tested without a database. The approach is to create a disposable test database, seed it with known data, and record what the functions return.

Test Database Setup

// tests/Integration/DatabaseTestCase.php
<?php
use PHPUnit\Framework\TestCase;

abstract class DatabaseTestCase extends TestCase
{
    protected static $connection;
    protected static $prefix = "test_";

    public static function setUpBeforeClass(): void
    {
        // Connect to a test database (NEVER production)
        self::$connection = mysqli_connect(
            'localhost', 'test_user', 'test_pass', 'bcoem_test'
        );

        // Load the schema from the install script
        $schema = file_get_contents(ROOT . 'setup/install_db.setup.php');
        // ... or use a SQL dump

        // Make $connection available globally
        // (this is how common.lib.php expects it)
        $GLOBALS['connection'] = self::$connection;
        $GLOBALS['prefix'] = self::$prefix;
        $GLOBALS['database'] = 'bcoem_test';
    }

    protected function setUp(): void
    {
        // Wrap each test in a transaction and roll back
        mysqli_begin_transaction(self::$connection);
    }

    protected function tearDown(): void
    {
        mysqli_rollback(self::$connection);
    }
}

Example Test

// tests/Integration/TotalFeesTest.php
<?php
class TotalFeesTest extends DatabaseTestCase
{
    public function test_total_fees_no_discount_single_user(): void
    {
        // Seed: 1 user with 3 entries at $8/each, no discounts
        $this->seedUser('user1', 3, brewerDiscount: 'N');

        $result = total_fees(
            entry_fee: 8.00,
            entry_fee_discount: 0,
            entry_discount: 'N',
            entry_discount_number: 0,
            cap_no: 0,
            special_discount_number: '',
            bid: 'default',
            filter: 'default',
            comp_id: 1
        );

        // Record what it ACTUALLY returns — don't assume 24.00
        $this->assertSame(24.00, $result);
    }

    public function test_total_fees_with_volume_discount(): void
    {
        // Seed: 1 user with 5 entries, discount kicks in after 3
        $this->seedUser('user2', 5, brewerDiscount: 'N');

        $result = total_fees(
            entry_fee: 8.00,
            entry_fee_discount: 6.00,
            entry_discount: 'Y',
            entry_discount_number: 3,
            cap_no: 0,
            special_discount_number: '',
            bid: 'default',
            filter: 'default',
            comp_id: 1
        );

        // 3 × $8 + 2 × $6 = $36 — but verify against actual output
        $this->assertSame(36.00, $result);
    }

    public function test_total_fees_with_cap(): void
    {
        // Seed: fees that exceed the cap
        $this->seedUser('user3', 10, brewerDiscount: 'N');

        $result = total_fees(8.00, 0, 'N', 0, 50, '', 'default', 'default', 1);

        // 10 × $8 = $80, capped at $50
        $this->assertSame(50.00, $result);
    }
}
Critical: the caret-delimited return values Several functions (brewer_info(), get_table_info(), currency_info()) return data as caret-delimited strings that callers explode("^", ...) and index by position. Your characterization tests must capture the exact string, including field order. If a refactor reorders the fields, the test must fail.

5. Tier 3 — Approval Tests for default.admin.php

Hardest — HTML Snapshot Testing

admin/default.admin.php is a 3,106-line procedural script that produces HTML. It has no functions to call — it's a single rendering pass driven by session state and recordsets. You can't unit-test it. The technique here is approval testing: capture the full HTML output under known conditions and compare it on each run.

How Approval Testing Works

Set up known state

Seed a test database, populate $_SESSION with a fixed set of values, and define all the global variables the file expects ($base_url, $go, $judging_started, etc.).

Capture output with output buffering
ob_start();
include ROOT . 'admin/default.admin.php';
$html = ob_get_clean();

// Strip volatile parts (timestamps, CSRF tokens)
$html = preg_replace('/\?t=\d+/', '?t=TIMESTAMP', $html);

// Save as the "approved" baseline
file_put_contents('tests/Approval/default_admin_full.approved.html', $html);
On future runs, compare against the baseline
$this->assertStringEqualsFile(
    'tests/Approval/default_admin_full.approved.html',
    $html
);

If the output changes, the test fails and the diff shows you exactly what shifted.

6. Recommended Sequence

Phase 1: Immediate wins (1-2 days)

Install PHPUnit. Write characterization tests for the ~25 pure functions in Tier 1. This covers ~1,800 lines and gives you a safe extraction path for splitting common.lib.php into focused files.

Phase 2: Fee & scoring safety net (3-5 days)

Set up a test database. Write integration tests for total_fees(), total_fees_paid(), best_brewer_points(), and brewer_info(). These are the highest-risk functions — they handle money and competitive results.

Phase 3: The big rendering functions (1 week)

Characterize style_convert() (7 modes), get_table_info() (6 modes), and data_integrity_check(). These are the longest functions and the ones most likely to contain subtle conditional bugs.

Phase 4: Admin dashboard approval tests (3-5 days)

Build the approval test harness for default.admin.php. Create 4-5 scenario snapshots. Once these pass, you can begin extracting sections of the 3,106-line file into smaller partials.

Phase 5: Ongoing extraction

With the safety net in place, begin the actual refactoring: extract pure functions into dedicated files, wrap database calls in repository classes, separate HTML from logic in the section files. Each extraction should be a small commit that passes all existing characterization tests.

The end goal You're not trying to achieve 100% code coverage. You're trying to put enough characterization tests around the two most dangerous files (combined: 8,583 lines, 2,865 decision points, 462 commits of churn) that you can safely break them apart without breaking the application. 60-70% coverage of these two files is worth more than 90% coverage of everything else.
